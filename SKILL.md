# Coding Workflow Skill

このスキルは、コーディング作業を体系的に進めるための構造化されたワークフローを実装します。ユーザーの指示を受けて、設計書作成→レビュー→TODO作成→レビュー→実装という5段階のプロセスを実行します。

## ワークフロー概要

コーディング作業は以下の7つのステップで進行します：

1. **ユーザー指示受領**: ユーザーからのコーディング指示を確認
2. **設計書作成**: markdown形式で設計書を作成またはupdate
3. **【ゲート1】設計レビュー**: ユーザーに設計内容のレビューを依頼
4. **TODO作成**: 具体的な実装ステップのTODOリストを作成
5. **【ゲート2】TODO承認**: ユーザーにTODOリストの実装承認を依頼
6. **実装**: ユーザー承認後に実装を開始
7. **ファイル一覧**: 作成・変更したファイル一覧を提示

## 詳細手順

### ステップ2：設計書作成

設計書は以下の構成を含める必要があります：

```markdown
# [タイトル]

## 概要
[作業の目的と背景を記述]

## 要件
[実装する機能・解決すべき課題を記述]

## 設計方針
[実装方法、ファイル構成、API設計など]

## その他
[考慮事項、リスク、後方互換性など]
```

### ステップ3：【ゲート1】設計レビュー

設計書を表示した直後に、以下の形式でレビュー依頼を出力します：

```
【ゲート1】設計書レビュー
上記の設計方針をご確認ください。
問題がなければ「承認」とお答えください。修正が必要な場合は「修正」とお答えください。
```

**ユーザーから承認を得るまで、次のステップに進みません。**

### ステップ4：TODO作成

TODOリストはTodoWrite toolを使用して作成します。以下の形式で3項目以上のタスクを含めます：

```
## TODO リスト

- [ ] タスク1: [説明]
- [ ] タスク2: [説明]
- [ ] タスク3: [説明]
```

### ステップ5：【ゲート2】TODO承認

TODOリストを表示した直後に、以下の形式で承認依頼を出力します：

```
【ゲート2】TODO承認
上記のTODOリストで実装を進めてよいでしょうか？
「承認」とお答えいただければ、実装を開始します。
```

**ユーザーから承認を得るまで、実装に進みません。**

### ステップ6：実装

ユーザーの承認後、以下のプロセスで実装を進めます：

1. TODO項目ごとに実装を進める
2. 実装完了時にTODOを✓で完了マーク
3. エラーが発生した場合は、デバッグ結果をユーザーに報告

### ステップ7：ファイル一覧表示

実装完了後、以下の形式で作成・変更したファイル一覧を表示します：

```markdown
## 作成・変更ファイル一覧

| ファイル名 | 説明 | ステータス |
|---------|------|--------|
| file1.ts | [説明] | 作成 |
| file2.ts | [説明] | 変更 |
```

## 重要なルール

### ゲート厳密実装
- **【ゲート1】【ゲート2】は必ず出力する** - これらのゲートをスキップしてはいけません
- ユーザーが明示的に「承認」と言うまで、次のステップに進まない
- ゲートメッセージは必ず独立したセクションとして、ユーザーに対する明示的な問いかけを含める

### 設計書フォーマット
- markdown形式で作成
- ファイル名は `DESIGN.md`
- 概要、要件、設計方針は必須

### TODO管理
- TodoWrite toolを使用して管理
- 実装中にTODOを更新し、完了時に✓マークを付与
- 具体的で実行可能なタスクに分割

## 実装例

### ユーザー指示
「新規ユーザー登録APIを実装してください。メールアドレスとパスワードで登録できるようにしてください」

### ステップ2出力（設計書）
```markdown
# ユーザー登録API実装

## 概要
メールアドレスとパスワードによるユーザー登録API（POST /api/users/register）を実装する

## 要件
- メールアドレスの重複チェック
- パスワードのハッシュ化と保存
- 登録成功時に201 Createdを返す
- メール形式の検証

## 設計方針
- エンドポイント: POST /api/users/register
- リクエストボディ: { email: string, password: string }
- レスポンス: { id: string, email: string }
- パスワードハッシュ: bcrypt を使用
```

### ステップ3出力
```
【ゲート1】設計書レビュー

上記の設計方針をご確認ください。
問題がなければ「承認」とお答えください。修正が必要な場合は「修正」とお答えください。
```

### ステップ4出力
```
## TODO リスト

- [ ] TODO-01: API エンドポイント定義（routes/users.ts）
- [ ] TODO-02: メール検証ロジック実装
- [ ] TODO-03: パスワードハッシュ化処理実装
- [ ] TODO-04: ユーザーモデル拡張
- [ ] TODO-05: エラーハンドリング実装
```

### ステップ5出力
```
【ゲート2】TODO承認

上記のTODOリストで実装を進めてよいでしょうか？
「承認」とお答えいただければ、実装を開始します。
```

## 注記

このスキルは、コード品質を確保し、ユーザーの意思決定ポイントを明確にするために設計されています。2つのゲート（設計レビューとTODO承認）を必ず実装することで、見落としや誤った方向性での実装を防ぐことができます。
